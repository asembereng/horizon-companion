Chapter 4 & 5: Clean Code & Design Principles — Domain Models

📁 horizon/models.py · horizon/services.py

Transforms the simple dictionary approach into proper domain models using dataclasses, enums, and the SOLID principles. Introduces the Repository pattern and dependency injection.

Python horizon/models.py
@dataclass
class Task:
    """A Horizon task — SRP: holds task data only."""
    id: int
    title: str
    assignee: User
    status: TaskStatus = TaskStatus.TO_DO
    priority: TaskPriority = TaskPriority.NORMAL

    def can_be_deleted_by(self, user: User) -> bool:
        """Only the assignee can delete a task."""
        return user.id == self.assignee.id
Python horizon/models.py — Dependency Inversion
class TaskRepository(Protocol):
    """Abstract interface — depend on this, not a concrete DB."""
    def save(self, task: Task) -> Task: ...
    def find_by_id(self, task_id: int) -> Task | None: ...
    def delete(self, task_id: int) -> bool: ...
SOLID in action: Task holds data (SRP). TaskRepository is a Protocol — an abstract interface (DIP). TaskService depends on the interface, not a specific database. You can swap SQLite for PostgreSQL without changing a single line in the service layer.
Dataclasses Enums SRP OCP DIP Protocol/Interface Repository pattern

Chapter 5: Service Layer — Coordinating Operations

📁 horizon/services.py

The service layer coordinates operations between models, repositories, and notification services. Demonstrates dependency injection — the service receives its dependencies, it doesn't create them.

Python horizon/services.py
class TaskService:
    def __init__(self, repo: TaskRepository, notifier: NotificationService | None = None):
        self._repo = repo        # Injected — could be in-memory or SQL
        self._notifier = notifier # Injected — could be console or email

    def delete_task(self, task_id: int, requesting_user: User) -> bool:
        """Delete a task IF the user has permission."""
        task = self._repo.find_by_id(task_id)
        if not task.can_be_deleted_by(requesting_user):
            raise PermissionError("Not authorized")
        return self._repo.delete(task_id)
💡 Key insight: Notice TaskService.__init__ takes repo and notifier as parameters. This is dependency injection. In tests, you pass InMemoryTaskRepository. In production, you pass SQLiteTaskRepository. The service doesn't know or care which one — it just calls .save().

Chapter 6: Data Modeling — SQLite Database

📁 horizon/database.py · tests/test_ch06_database.py

Real database persistence using SQLite. Demonstrates SQL schema design, parameterised queries (preventing SQL injection), and the repository pattern with a concrete implementation.

SQL Schema (inside database.py)
CREATE TABLE IF NOT EXISTS tasks (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    title       TEXT    NOT NULL,
    status      TEXT    NOT NULL DEFAULT 'to_do',
    priority    INTEGER NOT NULL DEFAULT 1,
    assignee_id INTEGER NOT NULL REFERENCES users(id),
    project_id  INTEGER REFERENCES projects(id),
    created_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);
⚠️ Security note: All queries use ? placeholders — never f-strings or concatenation. conn.execute("SELECT * FROM tasks WHERE id = ?", (task_id,)) prevents SQL injection attacks. This is non-negotiable in production code.

Chapter 10: Security — Authentication & Defence in Depth

📁 horizon/auth.py · tests/test_ch10_security.py

Python horizon/auth.py — Token Verification
def verify_token(token: str) -> dict | None:
    """Verify a token's signature and expiration."""
    parts = token.split(".")
    if len(parts) != 3: return None

    header_b64, payload_b64, signature = parts
    expected = hmac.new(_SECRET_KEY.encode(),
        f"{header_b64}.{payload_b64}".encode(), hashlib.sha256).hexdigest()

    if not hmac.compare_digest(signature, expected):
        return None  # Tampered!
⚠️ Production note: This is a simplified JWT implementation for learning. In production, use a library like PyJWT or python-jose. The concepts (signing, verification, expiration) are identical — the library handles edge cases.

class HorizonError(Exception):
    """Base exception — catch all Horizon errors with one line."""

class ValidationError(HorizonError): ...
class NotFoundError(HorizonError): ...
class AuthenticationError(HorizonError): ...
class AuthorizationError(HorizonError): ...
Why custom exceptions? Instead of bare raise Exception("not found"), you raise NotFoundError("Task", 42). This gives callers specific error types to catch, includes structured data (resource, identifier), and makes debugging trivial — the error tells you what wasn't found and which ID was missing.

Chapter 13: Containers — Dockerfile

📁 Dockerfile

Packages Horizon into a Docker container with layer caching, health checks, and a slim base image. Run anywhere with: docker build -t horizon . && docker run -p 8000:8000 horizon

Dockerfile Dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY horizon/ ./horizon/
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=5s \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"
CMD ["python", "-m", "horizon.api"]
💡 Layer caching: Notice requirements.txt is copied and installed before the application code. Docker caches each layer — so if only your code changes (not dependencies), the pip install step is cached and the rebuild is instant.
